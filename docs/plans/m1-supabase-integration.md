5. A real PUT to that signed URL lands a file in the `flight-recall-media` bucket (verified live).
6. `/sessions` on the live Vercel URL renders DB-backed data.

## Known M1 debt — must address in M4

We accumulated **two separate** pieces of security debt during M1, both deliberately. **Both must be undone together in M4. Re-enabling RLS while leaving the broad grants in place will not actually secure the tables.**

### Debt 1 — RLS disabled on three tables

During local verification, Supabase had **auto-enabled Row Level Security** on `public.aircraft`, `public.preflight_sessions`, and `public.media_assets` despite the migration script not containing any `enable row level security` statements. This is consistent with Supabase's newer project defaults (post the publishable-key revamp): RLS is forced on for new tables in the `public` schema. The publishable key uses the `anon` role, which had no policies, so every read returned an empty array and every write returned `42501 — new row violates row-level security policy`.

As an explicit M1 deferral, we disabled RLS on all three tables (run by the project owner in the Supabase SQL Editor; the agent did not touch this — RLS toggling is a production-security-relevant change that stayed under human control):

```sql
alter table public.aircraft            disable row level security;
alter table public.preflight_sessions  disable row level security;
alter table public.media_assets        disable row level security;
```

### Debt 2 — Broad anon GRANTs on three tables

Even after disabling RLS, the publishable key was still being rejected. Diagnostic queries showed Supabase's new Data API path requires explicit `GRANT`s to the `anon` role to be visible (the legacy implicit grants are not the same in the publishable-key world). To unblock M1 we ran:

```sql
grant select, insert, update, delete on public.aircraft            to anon;
grant select, insert, update, delete on public.preflight_sessions  to anon;
grant select, insert, update, delete on public.media_assets        to anon;
```

Diagnostic output captured at the moment of the fix (kept here for the record):

> `pg_tables.rowsecurity` was `true` on all three tables (pre-disable).
> `anon` and `authenticated` already had `SELECT, INSERT, UPDATE, DELETE, REFERENCES, TRIGGER, TRUNCATE` on all three tables. The `GRANT` statements were therefore no-ops, but were run as belt-and-suspenders.

### Why this is dangerous if not undone

Right now, anyone who can read the value of `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (which is shipped to every browser that loads the app) can read, insert, update, or delete every row in those three tables. There is **no auth boundary**. This is acceptable for M1 demo because there is no real pilot data and no user accounts — but it must not ship to a real pilot user.

### Debt 5 — Base GRANT on `storage.buckets`

`anon` was granted `SELECT` on `storage.buckets` so the bucket-read RLS policy from Debt 4 could actually be evaluated. This matches Supabase's default broad-grant pattern (the same grants exist for `authenticated` and `service_role` — Supabase ships storage with permissive grants and relies on RLS as the real security layer). M4 should evaluate whether to revoke `anon`'s grant entirely (and re-add via auth-scoped policy logic) or leave it and rely on stricter RLS policies. Either is defensible; the call should be made together with the rest of the Storage hardening.

### Debt 6 — `flight-recall-media` bucket created via SQL insert, not the dashboard

The dashboard's "Create bucket" flow appeared to succeed but never actually persisted a row in `storage.buckets`. We discovered this only after exhausting RLS / GRANT / key-format hypotheses — every "Bucket not found" error was Supabase telling the truth. The bucket was eventually created via a direct `insert into storage.buckets` and verified to have `id = 'flight-recall-media'`, `public = false`, matching what Debts 3/4/5 already targeted. No further policy work was needed.

**Implication for M4:** evaluate moving bucket creation into version-controlled SQL migrations (e.g., a top-of-file `insert into storage.buckets … on conflict do nothing` in `0001_m1_schema.sql` going forward, or its M4 successor) rather than relying on the dashboard. SQL-managed buckets are reproducible across environments; dashboard-created buckets are an out-of-band step that drift across staging/prod and can silently fail (as this one did).

**Note on the env var:** `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` currently holds a legacy JWT anon key (`eyJ…` format) rather than the new `sb_publishable_*` token. We swapped in the JWT during the bucket-debugging dead-end on the (incorrect) hypothesis that publishable keys had incomplete Storage support. The actual blocker turned out to be the missing bucket, not the key format. We're leaving the JWT in place for M1 because the keys are functionally equivalent for our anon-role use case and another swap means another redeploy. M4's auth refactor moves storage to user-token access anyway, at which point the anon-key choice becomes irrelevant.

### M4 hard prerequisite — all six debts must be undone, atomically

A new migration `supabase/migrations/0002_m4_rls_policies.sql` (plus a Storage cleanup script and an env-var swap) must:

1. **Revoke the broad table grants** to `anon` (Debt 2) before re-enabling RLS — grants override RLS policies, so leaving them in place would render the policies meaningless:
   ```sql
   revoke select, insert, update, delete on public.aircraft            from anon;
   revoke select, insert, update, delete on public.preflight_sessions  from anon;
   revoke select, insert, update, delete on public.media_assets        from anon;
   ```
2. **Re-enable table RLS** (Debt 1):
   ```sql
   alter table public.aircraft            enable row level security;
   alter table public.preflight_sessions  enable row level security;
   alter table public.media_assets        enable row level security;
   ```
3. **Add table policies** scoped to `auth.uid()`. At minimum:
   - `aircraft`: read for any authenticated user (or scope by `owner_id` if/when multi-tenant aircraft is on the table).
   - `preflight_sessions`: select/insert/update where `created_by = auth.uid()` (column to be added in M4).
   - `media_assets`: select/insert where the joined `preflight_sessions.created_by = auth.uid()`.
4. **Drop the M1 storage policies** (Debts 3 + 4) — `grep m1_anon_` to find them all:
   ```sql
   drop policy "m1_anon_upload_flight_recall_media"     on storage.objects;
   drop policy "m1_anon_read_flight_recall_media"       on storage.objects;
   drop policy "m1_anon_update_flight_recall_media"     on storage.objects;
   drop policy "m1_anon_read_flight_recall_media_bucket" on storage.buckets;
   ```
5. **Re-evaluate Debt 5** (`anon` SELECT on `storage.buckets`). Either revoke or leave with a documented justification.
6. **Add `flight-recall-media` storage policies scoped to `auth.uid()`** — read/write only for objects owned by the authenticated user (typically by namespacing the storage_key with the user UUID and matching `(storage.foldername(name))[1] = auth.uid()::text`).
7. **Swap the env var** (Debt 6) — re-evaluate `sb_publishable_*` Storage support; if fully supported by then, swap. Otherwise leave the legacy JWT in place but understand that M4 auth flows attach the user's JWT and override anon entirely.

**This is a hard prerequisite for M4, not a nice-to-have.** Re-enabling RLS without revoking grants and dropping permissive policies leaves the tables wide open. All six debts move together.

**Tracking:** This debt is owned by whoever picks up M4. The migration file (`supabase/migrations/0001_m1_schema.sql`) has a comment explaining the RLS deferral; M4's `0002_m4_rls_policies.sql` is where these debts get repaid in one atomic apply.
